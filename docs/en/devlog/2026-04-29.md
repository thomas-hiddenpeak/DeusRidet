# 2026-04-29 — Auditus Upstream Python Smoke

## Step 18l — ClearerVoice + 3D-Speaker + FunASR isolated smoke

### Context

Thomas proposed a deliberately sideways probe: test the original Python
environments from the three upstream projects directly, but preserve the
custom Orin PyTorch stack. The critical constraint was that `py313` must not
be polluted or replaced by online torch wheels; a CPU torch result means the
environment has already been damaged.

### Approach

Created an isolated conda clone named `py313_audio_labs` from `py313`, then
used `uv pip --python /home/rm01/miniconda3/envs/py313_audio_labs/bin/python`
with `--no-deps` for source packages and surgical non-torch dependencies.
Full upstream requirements were not installed because they contain unsafe
torch-family and NumPy pins for this platform.

Upstream repositories were checked out under `/home/rm01/upstream-audio-labs`:

- ClearerVoice-Studio: `6b3774d`
- 3D-Speaker: `065629c`
- FunASR: `b842ff8`

After installation and cleanup, both environments still reported the intended
CUDA torch baseline:

- `py313`: `torch 2.10.0`, CUDA available, CUDA 12.6.
- `py313_audio_labs`: `torch 2.10.0`, CUDA available, CUDA 12.6.

### Result

ClearerVoice-Studio smoke passed with `MossFormer2_SS_16K` on the bundled
`input_ss.wav`. It produced two 16 kHz mono separated files, each 4.409 s:

- `/home/rm01/upstream-audio-labs/results/clearvoice_ss_smoke/output_MossFormer2_SS_16K_s1.wav`
- `/home/rm01/upstream-audio-labs/results/clearvoice_ss_smoke/output_MossFormer2_SS_16K_s2.wav`

ClearerVoice printed a GPU-selection warning while parsing a Jetson-style
`[N/A]` field, but torch CUDA remained functional and the run completed.

3D-Speaker CAM++ speaker verification passed with
`iic/speech_campplus_sv_zh-cn_16k-common`. The upstream script reported GPU
inference, wrote two `(192,)` embeddings, and produced similarity `0.4807`.
The smoke used a process-local `soundfile` loader and
`torchaudio.functional.resample` shim because torchaudio 2.10's raw
`torchaudio.load` path wants TorchCodec, and the PyPI TorchCodec wheel did not
load cleanly on this stack.

FunASR SenseVoiceSmall smoke passed on CUDA with `fsmn-vad`. Initialization
took 46.723 s, inference took 1.107 s, and the recognized text was:

> The tribal chieftain called for the boy and presented him with 50 pieces of gold.

The consolidated smoke record is stored at
`/home/rm01/upstream-audio-labs/results/smoke_summary.md`.

### Conclusion

The upstream Python stacks are usable for controlled experiments on Orin as
long as they are run from the isolated `py313_audio_labs` environment with
dependency resolution constrained. They should not be installed wholesale into
the production `py313` environment. The next useful experiment is to run these
upstream chains on DeusRidet's high-risk Auditus clips and score them with the
existing GT identity monitor.

## Step 18m — Upstream-vs-C++ Auditus pipeline probe

### Context

Thomas asked to use the original upstream methods on the real Auditus data and
pipeline, primarily to determine whether the low separated-source identity
ratio points to a DeusRidet C++ program bug.

### Approach

Added `tools/auditus_upstream_audio_probe.py`, a repeatable probe that runs the
same top-30 high-risk homogeneity clips through two separator variants:

- `upstream_clearvoice`: original Python ClearVoice `MossFormer2_SS_16K`, with
  only a process-local GPU-selector shim for Jetson/Orin `nvidia-smi` parsing.
- `cpp_raw_o16000`: existing DeusRidet C++ raw_o16000 separation output.

Both variants are evaluated by the same original downstream stack in the
isolated `py313_audio_labs` environment: FunASR SenseVoiceSmall for ASR and
3D-Speaker CAM++ for speaker evidence. The GT alignment reuses the existing
separated-source ASR and identity summarizers.

### Result

Artifacts:

- `logs/upstream_audio_probe_r1/upstream_vs_cpp_comparison.md`
- `logs/upstream_audio_probe_r1/upstream_vs_cpp_diagnosis.md`
- `logs/upstream_audio_probe_r1/upstream_clearvoice/`
- `logs/upstream_audio_probe_r1/cpp_raw_o16000/`

Same downstream comparison:

| Variant | ASR Two | ASR One | ID Two | ID One | Correct | Wrong | Abstain | Timeline |
|---------|--------:|--------:|-------:|-------:|--------:|------:|--------:|---------:|
| upstream_clearvoice | 6 / 30 | 30 / 30 | 0 / 30 | 20 / 30 | 20 | 7 | 14 | 20 / 49 = 0.408 |
| cpp_raw_o16000 | 6 / 30 | 30 / 30 | 2 / 30 | 19 / 30 | 21 | 7 | 12 | 21 / 49 = 0.429 |

Rank deltas are narrow: upstream-only ASR two-speaker rank is 4; C++-only ASR
two-speaker rank is 16. C++-only identity two-speaker ranks are 5 and 13.

Waveform comparison shows the primary sources are very close (`src1` mean
correlation 0.974). Weak residual sources differ more (`src2` mean correlation
0.734, p10 0.294). Switching the C++ output to `audio_official_rms` changes RMS
but not correlation, so the weak-source shape difference is real; however, it
does not translate into better upstream identity metrics.

The earlier DeusRidet C++ downstream on the same `cpp_raw_o16000` separation
still scores higher than the original Python downstream: Qwen+Dual recovered
24 / 49 GT rows with 3 two-ID clips, versus FunASR+3D-Speaker's 21 / 49 and 2
two-ID clips on the same C++ separator output.

### Conclusion

The original upstream Python stack does not implicate a broad C++ separator
bug. The current bottleneck is source-level identity evidence on short or weak
separated streams, plus a few per-rank weak-source shape differences. The next
engineering target should be runtime selector/evidence fusion and weak-source
handling, not replacing the C++ separator wholesale.
