## Why this step exists

The philosophical evaluation of the hearing subsystem (this morning's
conversation with Thomas) produced eight claims ranging from tier
alignment to "allow contradictions / judge holistically". All of them
collapse to one operational question: *does the Auditus pipeline
actually hear what humans hear?* A numeric WER script cannot answer
that — per `benchmarks.instructions.md` **R1**, human-style reading is
the only acceptable judging method. Step 15 builds the minimum
substrate that lets an intelligent reader (Thomas, or the agent) look
at a side-by-side timeline of ground truth and pipeline output and
write honest findings.

The substrate has three pieces:

1. **Canonical T0 ground truth** derived once from `tests/test.txt`.
2. **A headless capture tool** that replays an audio file through the
   running `awaken` WebSocket and records every `asr_transcript`
   broadcast in chronological `hyp.jsonl`.
3. **A judge document generator** that merges GT and HYP into a single
   Markdown, bucketed by one-minute windows, for a human-or-agent
   reader to mark up.

No WER. No speaker-accuracy percentage. Just a document the reader can
actually read.

## The three bugs found during the end-to-end dry run

Running it proved the only three things that could possibly be wrong:

### Bug 1 — nested f-string with backslash escapes

`capture_auditus_run.py` originally interpolated a `rec["speaker_name"]`
fallback inside a print-format f-string and tried to escape inner
quotes with `\"`. Python 3.10 parses the backslash before the format
spec, so the file refused to import. Fix: hoist a plain
`spk_label = rec["speaker_name"] or f"spk{rec['speaker_id']}"` line
before the print. Lesson: if an f-string needs more than one layer of
substitution, it's begging to become a local variable.

### Bug 2 — server ASR is off by default

`DEUSRIDET_TEST_WS_ENABLE_ASR=1 ./build/deusridet awaken` only toggles
the *model load*; `enable_asr_` on the pipeline itself defaults to
`false` because speaker-only benchmarking is the current dev mode (see
`src/actus/awaken_router.cpp:312` — the router accepts an
`asr_enable:on` text message). Without that flag the pipeline emits no
`asr_transcript` broadcasts at all. The capture tool now sends
`asr_enable:on` immediately after WS connect, matching the contract
the router documents for the WebUI.

### Bug 3 — 4× speed saturates the pipeline

The first full-length capture at `--speed 4.0` fed the entire 3615 s
file through WS in 904 s wall time, then drained for 60 s. Raw event
counts: 96 303 total frames, of which **66 392 were `audio_drop`**.
Only 353 `asr_transcript` events were emitted, and `hyp.jsonl` ends at
T0 1509 s — coverage stops at **41.8 %** of the source audio.

This is not a bug in the capture tool. It is a real property of the
pipeline: somewhere downstream of the WS ingest, the ring buffer
cannot absorb 4× real-time audio while ASR + speaker-ID + VAD are all
running. The drop happens silently from the WS perspective. The
lesson is exactly what `benchmarks.instructions.md` warned about:
*lower the speed if the backend cannot keep up*. A `--speed 1.0` run
is underway in parallel and the appendix below will record its
coverage, latency, and judge findings.

## First-pass findings from the 4× capture

Even with only 41.8 % coverage the first pass reveals several strong
signals the slower run is very unlikely to reverse.

### A. Speaker tracking collapses four humans into one ID

`tests/test.txt` has four distinct speakers (朱杰, 徐子景, 石一,
唐云峰). In the 4× run:

- 231 / 353 HYP rows (65.4 %) are `spk0`.
- 89 (25.2 %) are `spk1`.
- 24 (6.8 %) are `spk2`.
- 9 (2.5 %) are `spk=-1` (unknown / pre-embedding).

In the first ~10 minutes **every** attributed utterance is `spk0`
regardless of which human actually spoke. Example at T0 78–89 s:

| GT (t0) | Speaker | Text (GT) |
|--------:|:--------|:----------|
| 78 | 唐云峰 | 对，跟成都没关系。 |
| 82 | 朱杰 | 然后呢， |
| 83 | 唐云峰 | 15的话，那我5%，我可以的... |
| 88 | 朱杰 | 我还没说完。 |

All four lines come back attributed to `spk0` by the pipeline. This
is not a threshold tuning problem (raising/lowering `m_thresh`
won't help when four voices collapse into one centroid); it is a
tracker-registration / re-ID problem. Candidate causes:

1. The SpeakerTracker starts with an empty DB and accepts the first
   sustained embedding as `spk0`; subsequent short, overlapping turns
   never clear the registration gate (`pair stable`, `Registration
   gate failed` log lines appear dozens of times per minute).
2. Four-person conversation at meeting pace produces many utterances
   under the minimum-duration gate; only the longest survive, and
   they are dominated by one speaker (朱杰 in the first ten minutes).
3. The CAM++/WL-ECAPA embeddings are being computed on the FRCRN /
   MossFormer2 denoised signal, which attenuates low-energy speakers.
   (This is testable — disabling the denoise chain is one of the
   eight philosophical claims.)

### B. ASR transcription is unexpectedly decent

When the pipeline does emit text, the Chinese transcription is often
semantically correct on long, clear utterances. Representative:

| GT | HYP | Verdict |
|----|-----|---------|
| 不要铺垫，有话直说。你要是以后铺垫，你就当你最前面说的话为准。 | 不要铺垫，有话直说。你要是以后铺垫，就当你最前面说的话为准。 | ✅ semantic match, one dropped pronoun |
| 你不知道我说话喜欢铺垫吗。 | 你知道我说话喜欢铺垫吗。 | ❌ negation dropped |
| 拿出40%的精力 | 拿不出百分之四十的精力 | ❌ hallucinated negation + digit→character |
| 15的话，那我5%，我可以的，只要超过3.14我就可以接受。 | 那我百分之5我可以的，只要超过三点一四，我就可以接受。 | ⚠️ numbers inconsistent (one in Arabic, one spelled) |
| 光速 (a VC firm) → 光远 (homophone, later corrected) | — | n/a (GT itself captures the human ambiguity) |

A recurring artefact: the token `language Chinese` occasionally leaks
into the output (`"因为。language Chinese因为。"`). That is a Whisper/
Qwen-ASR prompt-token bleeding into the decoded text — the decoder is
emitting the language-ID control token as literal text. That is
fixable in the ASR postprocessor, not in the model.

### C. Segmentation is finer than human annotation

GT turns are multi-minute for 朱杰's monologues; HYP segments them at
`post_silence` into 2–6 s chunks. Example: GT idx=17 (唐云峰, 108–141
s, 33 s long) comes back as **four** HYP rows covering substantially
the same content. Not a defect per se — post-silence is the right
segmentation policy for a continuous-consciousness engine — but it
means a future "merge HYP segments by speaker-run" step is needed
before any automated attribution metric could be computed. Good
reason not to chase one.

### D. Latency is healthy

| Metric | Value |
|--------|------:|
| p50 | 375 ms |
| p90 | 987 ms |
| max | 1781 ms |

These are measured as (ASR emit wall time) − (segment end wall time)
on the server side; at 4× replay speed they are inflated because the
pipeline is backlogged. The 1× run will give the true figure. Even
so, sub-second median on a 1.7 B ASR on a single Orin is already
good.
