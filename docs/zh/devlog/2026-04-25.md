## 这一步为何存在

今晨 Thomas 关于听觉子系统的八点哲学评估，其操作化结论其实只有一个：
*Auditus 管线到底听到了什么？* `benchmarks.instructions.md` 的 R1 明
文规定——数值脚本不能作判官，"人类式通读" 是唯一被接受的评判方式。
因此 Step 15 的任务是造出一份"判官能直接读的材料"，让一个有智慧的
读者（Thomas 或 agent）去看真话-pipeline 结果的并排时间线，并亲手
写下所见。

这份基座由三件东西组成：

1. **T0 对齐的真话（Ground Truth）** —— `tests/test.txt` 一次性解析
   为标准 JSON。
2. **无头捕获工具** —— 把音频回放进 `awaken` 的 WebSocket，把所有
   `asr_transcript` 广播按时间顺序写入 `hyp.jsonl`。
3. **判官文档生成器** —— 把 GT 与 HYP 按一分钟窗口合并为一份
   Markdown，供读者直接标注。

没有 WER、没有说话人准确率百分比，只有一份人（或 agent）可以真正
读下去的文档。

## 端到端 dry run 暴露的三个问题

把工具真正跑起来，三个可能出错的地方恰好全部踩中：

### 问题一 —— f-string 里套 f-string 还要转义引号

`capture_auditus_run.py` 原先在一条 print 的 f-string 内部试图用
`\"` 转义内层引号，取 `rec["speaker_name"]` 与 `f'spk{rec["speaker_id"]}'`
的回退。Python 3.10 在解析格式说明符之前就把反斜杠吃了，导致文件
直接无法 import。修法：把
`spk_label = rec["speaker_name"] or f"spk{rec['speaker_id']}"` 作为
单独一行前置。教训：只要 f-string 的嵌套多过一层，它就是在呼喊
"把我抽成局部变量"。

### 问题二 —— 服务器 ASR 默认关闭

`DEUSRIDET_TEST_WS_ENABLE_ASR=1 ./build/deusridet awaken` 只打开了
*模型加载*；管线本身的 `enable_asr_` 默认为 `false`，因为当前开发
节奏是"只做说话人基准"（见 `src/actus/awaken_router.cpp:312`：路由
接受 `asr_enable:on` 文本消息）。不发这条消息，管线就根本不广播
`asr_transcript`。捕获工具现在会在 WS 连通后立即发送
`asr_enable:on`，与路由器文档化的 WebUI 契约保持一致。

### 问题三 —— 4× 回放把管线挤爆

首次全长捕获设 `--speed 4.0`，把 3615 s 的音频在 904 s 墙钟内全部
喂进 WS，再排空 60 s。原始事件统计：96 303 条总帧，其中
**66 392 条是 `audio_drop`**。只有 353 条 `asr_transcript` 被发出，
`hyp.jsonl` 止于 T0 1509 s —— 覆盖率仅 **41.8%**。

这不是捕获工具的 bug，而是管线的真实属性：在 WS 摄入之下游的某个
环形缓冲区，无法在同时跑 ASR + 说话人 ID + VAD 的情况下吞下 4× 实
时音频。丢包从 WS 侧看是无声的。教训恰是
`benchmarks.instructions.md` 所警告的——*后端吃不下就降速*。
`--speed 1.0` 的实时跑法正在并行进行，结果将附于下方附录。

## 4× 首批结论

尽管只覆盖了 41.8%，数据已经给出几条 1× 实时跑极难翻盘的强信号。

### A. 说话人跟踪把四个人坍缩成一个 ID

`tests/test.txt` 有四位说话人（朱杰、徐子景、石一、唐云峰）。4× 跑
的结果：

- 231 / 353 HYP 行（65.4%）为 `spk0`
- 89（25.2%）为 `spk1`
- 24（6.8%）为 `spk2`
- 9（2.5%）为 `spk=-1`（未知 / 嵌入未成型）

前十分钟内，**每一条** 带说话人的 HYP 都是 `spk0`——不管 GT 是谁在
说。例如 T0 78–89 s：

| GT (t0) | 说话人 | 文本（GT） |
|--------:|:------|:----------|
| 78 | 唐云峰 | 对，跟成都没关系。 |
| 82 | 朱杰   | 然后呢， |
| 83 | 唐云峰 | 15的话，那我5%，我可以的⋯⋯ |
| 88 | 朱杰   | 我还没说完。 |

管线把这四条全部归为 `spk0`。这不是阈值问题（当四个人声压进一个
centroid 时 `m_thresh` 调高调低都无用），而是跟踪-注册/重识别问题。
可能原因：

1. SpeakerTracker 启动时 DB 为空，第一个持续嵌入被收为 `spk0`；后
   续短、重叠的话轮迟迟过不了注册门（日志里每分钟几十条 `pair
   stable`、`Registration gate failed`）。
2. 四人会议节奏下大量话轮被最小时长门砍掉，存活下来的长话轮前期几
   乎全是朱杰，数据倾斜。
3. CAM++/WL-ECAPA 嵌入是在 FRCRN/MossFormer2 去噪后的信号上算的，
   低能量说话人被压平。——这是八点哲学断言中的一条，可测：停掉去噪
   链直接对比。

### B. ASR 转写意外地不错

管线真有输出时，中文转写在长而清晰的话轮上语义基本对：

| GT | HYP | 评判 |
|----|-----|-----|
| 不要铺垫，有话直说。你要是以后铺垫，你就当你最前面说的话为准。 | 不要铺垫，有话直说。你要是以后铺垫，就当你最前面说的话为准。 | ✅ 语义一致，掉一个 "你" |
| 你不知道我说话喜欢铺垫吗。 | 你知道我说话喜欢铺垫吗。 | ❌ 否定丢失 |
| 拿出40%的精力 | 拿不出百分之四十的精力 | ❌ 幻觉出 "不"，数字变中文 |
| 15的话，那我5%，我可以的，只要超过3.14我就可以接受。 | 那我百分之5我可以的，只要超过三点一四，我就可以接受。 | ⚠️ 数字风格不统一 |
| 光速 / 光远 | — | 人本身就口误了 |

一处固定伪像：`language Chinese` 偶尔直接作为字面进入输出
（`"因为。language Chinese因为。"`）。这是 Whisper / Qwen-ASR 的
语种控制 token 从解码器里泄漏出来——应在 ASR 后处理器里屏蔽，不是
模型的问题。

### C. 分段比人工标注更细

GT 里朱杰的独白一段跨数分钟；HYP 在 `post_silence` 处切成 2–6 s
的小段。例：GT idx=17（唐云峰，108–141 s，33 s 一段）被切为 4 条
HYP，基本覆盖全部内容。这不是缺陷——"连续意识引擎"的正确分段策略
就是 post-silence——但意味着未来若想自动计归属分数，必须先做"按同
一说话人合并 HYP 片段"这一步。也正因如此，不该在此阶段追这个分数。

### D. 延迟健康

| 指标 | 数值 |
|------|----:|
| p50 | 375 ms |
| p90 | 987 ms |
| max | 1781 ms |

4× 回放时管线积压，这些数字是被放大的；1× 跑会给出真实值。即便如
此，单 Orin 跑 1.7 B ASR 的中位亚秒延迟，已经是漂亮的数字。

### 1× 仍未解决的混淆

多吞吐量消除了覆盖率问题，但没消除说话人归属问题：

1. **第一个说话人粘性**。GT idx 17（唐云峰，108–141 s，33 s 连
   续）在 1× 里仍被全部归给 `spk0`（朱杰），因为那会儿跟踪器还
   没注册第二个嵌入。4× 的行为一致。
2. **短插话漂移**。1537–1543 s 处 GT 徐子景说"雷总也不说话了。"
   HYP 把整段归给 `spk3`（石一）。当插话只有 ~2 s，跟踪器保留
   上一段说话人。
3. **重叠期 ASR 内容漂移**。1486–1500 s 石一列 VC（光速、天际、
   源点）时 HYP 转写部分失真（"啊，那时候工资还比较挺高"）。那
   段 GT 有多人同时说话（"光速跑了" / "哦，又是光速" / "最讨厌
   两家投资机构"）。重叠语音是 ASR 的硬场景，也是 HYP speaker ID
   在 `spk2` 与 `spk3` 间闪烁的地方。

### 1× 与 4× 共有的不变量

- `language Chinese` 控制 token 泄漏到输出文本——同一 t0 位置，
  说明这是确定性的后处理问题，不随速率变化。
- 数字风格不统一（"百分之四十" / "百分之5" / "三点一四"）在两种
  速度下都稳定存在——所以这是 Qwen3-ASR 解码的固有怪癖，不是压
  流效应。
- 否定丢失（"你不知道…" → "你知道…"）两次都出现——也是模型层
  问题，与管线饱和无关。

### 结论

1. **评测时降回实时回放速度。** 1× 跑是今后一切听觉子系统测量
   的 ground truth。4× 只适合作 **压力探针**——它可以量化"管线
   在 2× 以上某处饱和"；饱和点具体在哪值得专门做一次测量，但不
   是阻断项。
2. **说话人跟踪问题有两个独立部分**：(a) 冷启动代价（~5 分钟内
   才把 4 位说话人识齐），(b) 收敛后的短插话粘性。两者都是真实
   的、都在 ASR 关闭时仍存在、都与去噪路由无关。这是 Step 16
   该调研的。
3. **ASR 转写质量对于 1.7B 模型跑连续流来说可接受。** 可观察的
   错误（否定丢失、数字风格、控制 token 泄漏）都是模型或后处理
   问题，不是流或管线问题。后处理层加一个 `language Chinese`
   过滤器是一个小独立修复（Step 17 候选）。
4. **基座本身是对的。** 对用户只暴露两个参数：音频文件 + 回放速
   度。人类（或 agent）读的产物是 `judge.md`，由同样三件工具确
   定性生成，零自动打分。这正是 R1 想要的。
