# 2026-04-29 — Auditus 上游 Python Smoke

## Step 18l — ClearerVoice + 3D-Speaker + FunASR 隔离验证

### 背景

Thomas 提出一个更跳脱的探针：直接使用三个上游项目的原始 Python 环境测试，
但必须保护 Orin 上定制的 PyTorch 栈。关键约束是 `py313` 不能被在线 wheel
污染或替换；如果看到 CPU torch，就说明环境已经被破坏。

### 方法

从 `py313` 克隆出隔离 conda 环境 `py313_audio_labs`，之后只通过
`uv pip --python /home/rm01/miniconda3/envs/py313_audio_labs/bin/python`
安装，并对源码包使用 `--no-deps`，缺什么非 torch 依赖再逐个补。没有直接安装
任何上游完整 requirements，因为其中包含对 torch 家族和 NumPy 的不安全 pin，
不适合这台 Orin 的定制环境。

上游仓库检出在 `/home/rm01/upstream-audio-labs`：

- ClearerVoice-Studio：`6b3774d`
- 3D-Speaker：`065629c`
- FunASR：`b842ff8`

全部安装和清理后，两个环境都仍保持目标 CUDA torch 基线：

- `py313`：`torch 2.10.0`，CUDA available，CUDA 12.6。
- `py313_audio_labs`：`torch 2.10.0`，CUDA available，CUDA 12.6。

### 结果

ClearerVoice-Studio 使用 `MossFormer2_SS_16K` 对自带 `input_ss.wav` 的
smoke 通过，输出两路 16 kHz mono 分离音频，每路 4.409 s：

- `/home/rm01/upstream-audio-labs/results/clearvoice_ss_smoke/output_MossFormer2_SS_16K_s1.wav`
- `/home/rm01/upstream-audio-labs/results/clearvoice_ss_smoke/output_MossFormer2_SS_16K_s2.wav`

ClearerVoice 打印了一个 GPU 选择警告，原因是它解析 Jetson 风格的 `[N/A]`
字段失败；但 torch CUDA 本身正常，推理完成。

3D-Speaker 使用 `iic/speech_campplus_sv_zh-cn_16k-common` 的 CAM++ 说话人
验证 smoke 通过。上游脚本报告使用 GPU 推理，写出两份 `(192,)` embedding，
相似度为 `0.4807`。这次 smoke 使用了进程内 `soundfile` loader 和
`torchaudio.functional.resample` shim，因为 torchaudio 2.10 的原始
`torchaudio.load` 路径需要 TorchCodec，而 PyPI TorchCodec wheel 在当前栈上
动态库加载不干净。

FunASR 使用 SenseVoiceSmall + `fsmn-vad` 的 CUDA smoke 通过。初始化耗时
46.723 s，推理耗时 1.107 s，识别文本为：

> The tribal chieftain called for the boy and presented him with 50 pieces of gold.

汇总记录保存在 `/home/rm01/upstream-audio-labs/results/smoke_summary.md`。

### 结论

三个上游 Python 栈可以在 Orin 上用于受控实验，前提是运行在隔离的
`py313_audio_labs` 中，并严格约束依赖解析。它们不能整包安装进生产 `py313`。
下一步有价值的是把这些上游链路跑在 DeusRidet 的高风险 Auditus clip 上，
再接入已有 GT identity monitor 做同一套身份归因评分。

## Step 18m — 上游原装链路 vs C++ Auditus 管线探针

### 背景

Thomas 要求把上游原装方式直接用到真实 Auditus 数据和管线上，核心问题是：
分离源 identity 有效比例偏低，是否说明 DeusRidet 的 C++ 程序有问题。

### 方法

新增 `tools/auditus_upstream_audio_probe.py`，对同一批 top30 高风险
homogeneity clips 跑两个 separator variant：

- `upstream_clearvoice`：原装 Python ClearVoice `MossFormer2_SS_16K`，仅加
  一个进程内 GPU selector shim，绕过 Jetson/Orin 上 `nvidia-smi` 字段解析问题。
- `cpp_raw_o16000`：已有 DeusRidet C++ raw_o16000 分离输出。

两个 variant 使用完全相同的原装 downstream，在隔离 `py313_audio_labs` 中运行：
FunASR SenseVoiceSmall 做 ASR，3D-Speaker CAM++ 做说话人证据。GT 对齐复用
现有 separated-source ASR 与 identity summarizer。

### 结果

产物：

- `logs/upstream_audio_probe_r1/upstream_vs_cpp_comparison.md`
- `logs/upstream_audio_probe_r1/upstream_vs_cpp_diagnosis.md`
- `logs/upstream_audio_probe_r1/upstream_clearvoice/`
- `logs/upstream_audio_probe_r1/cpp_raw_o16000/`

同 downstream 对比：

| Variant | ASR 双说话人 | ASR 至少一路 | ID 双说话人 | ID 至少一路 | Correct | Wrong | Abstain | Timeline |
|---------|-------------:|-------------:|------------:|------------:|--------:|------:|--------:|---------:|
| upstream_clearvoice | 6 / 30 | 30 / 30 | 0 / 30 | 20 / 30 | 20 | 7 | 14 | 20 / 49 = 0.408 |
| cpp_raw_o16000 | 6 / 30 | 30 / 30 | 2 / 30 | 19 / 30 | 21 | 7 | 12 | 21 / 49 = 0.429 |

rank 差异很窄：上游独有 ASR 双说话人 rank 为 4；C++ 独有 ASR 双说话人 rank
为 16。C++ 独有 identity 双说话人 rank 为 5 和 13。

波形比较显示主源高度接近（`src1` 平均相关性 0.974），弱 residual 源差异更大
（`src2` 平均相关性 0.734，p10 为 0.294）。把 C++ 输出切到
`audio_official_rms` 只改变 RMS，不改变相关性，因此弱源形状差异是真实存在的；
但它没有转化为上游 identity 指标优势。

同一个 `cpp_raw_o16000` 分离输出上，早先 DeusRidet C++ downstream 仍然高于
原装 Python downstream：Qwen+Dual 恢复 24 / 49 个 GT rows、3 个 two-ID clips；
FunASR+3D-Speaker 在同一 C++ separator 输出上为 21 / 49、2 个 two-ID clips。

### 结论

原装上游 Python 链路没有指向“C++ separator 整体有 bug”。当前瓶颈更像是短、弱
分离源上的 source-level identity 证据不足，再加少数 rank 的弱源形状差异。下一步
应优先做 runtime selector / evidence fusion 和 weak-source handling，而不是整体替换
C++ separator。
