# 2026-04-30 — Auditus Fusion Selector

## Step 18n — 分离源身份侧证据融合

### 背景

上游原装链路 vs C++ 探针完成后，separator 本身已经不再是主要嫌疑。下一步目标是
把不同 variant 之间分散的证据转化成 runtime selector / fusion：选择时只使用 GT
评分前可见的运行时信号，之后再拿选中的 hypotheses 去对 GT 时间线打分。

基线 commit：`d235f96`。

### 方法

新增 `tools/auditus_separatio_fusion_selector.py`，这是一个离线 evaluator，读取 R1
和 R2 separator sweep 中已有的 C++ Qwen+Dual identity reports。selector 在选择证据
时不使用 GT 字段。运行时信号包括：

- 预测说话人身份；
- speaker similarity 与 margin；
- 分离源 RMS；
- 分离源 ASR 文本长度；
- 同一预测说话人的跨 variant 支持。

评估运行前执行了 benchmark 前置清理：杀掉 `deusridet` 进程、drop page cache，并用
`pgrep -x deusridet` 确认真正干净。

默认 fusion 规则保留 similarity `>= 0.35`、margin `>= 0.05`、ASR 文本非空的证据，
每个 clip 最多保留两个预测说话人。报告同时输出两种视角：

- `support_union`：被选中的 speaker hypothesis 保留所有合格的跨 variant 分离源证据。
- `single_best`：每个被选中 speaker 只用运行时分数最高的一条证据计分，作为保守下界。

### 结果

产物：

- `logs/separatio_fusion_selector_r1/fusion_selector_report.md`
- `logs/separatio_fusion_selector_r1/fusion_selector_report.json`

Top30 对比：

| Method | Two-ID | One-ID | Timeline | Correct Decisions | Wrong | Decision Acc |
|--------|-------:|-------:|---------:|------------------:|------:|-------------:|
| baseline raw_o16000 | 3 / 30 | 21 / 30 | 24 / 49 = 0.490 | 24 | 8 | 0.750 |
| fusion single_best | 3 / 30 | 22 / 30 | 25 / 49 = 0.510 | 25 | 7 | 0.781 |
| fusion support_union | 6 / 30 | 25 / 30 | 32 / 49 = 0.653 | 31 | 4 | 0.886 |

阈值扫描显示：当 margin 保持 `0.05` 时，similarity `0.30` 和 `0.35` 都能得到同样的
主结果；如果把 margin 降到 `0.00` 或 `0.03`，recall 不变，但 wrong decisions 从 4
升到 6。更严格的 similarity `0.40` 可以把 wrong decisions 降到 3，但 timeline recall
降为 29 / 49。

### 结论

真正有用的信号不是“某一个 separator 输出更好”，而是不同 separator variants 之间的
一致性与互补性。生产路径应保留同一预测说话人的多条高置信 source hypotheses，再在最终
timeline attribution 前融合 speaker identity 和 ASR 证据。下一步应把这个离线 selector
落成 Auditus runtime fusion stage，同时保持更严格的 margin guard，因为它明显降低了错误
说话人决策。

## Step 18o — Runtime fusion shadow stage 首次 live smoke

### 背景

离线 selector 需要一个不会扰动在线 speaker label 的 runtime 落脚点。因此第一阶段必须是
shadow-only：在 final ASR segment 之后运行，发出分离源证据，并持久化供复盘，但不触碰
`SpeakerTimeline` 或 `on_speaker_` 决策。

基线 commit：`d235f96`。

### 方法

新增由 `DEUSRIDET_AUDITUS_FUSION_SHADOW=1` 控制的 opt-in runtime 路径。final ASR
结果产生后，ASR worker 使用同一个 `ASRJob` PCM 运行已初始化的 MossFormer2 separator，
再把 `src1` / `src2` 送入现有 ASR engine 作为 shadow evidence。事件以
`stage=fusion_shadow` 的 `asr_log` payload 发出；Auditus facade 只把这个 payload 额外
写入 `TimelineLogger::log_fusion_shadow()`。

默认行为不变，也避免继续扩大已经逼近 R1 硬限制的 `audio_pipeline.h` 接口（499 行）。
这个路径在 runtime 可通过 WebSocket 观察，也会持久化到 timeline JSONL，但不改变 speaker
label。

## Step 18p — Runtime fusion shadow speaker evidence

### 背景

runtime shadow event 已经有了分离源 ASR 文本，但还缺生产 selector 需要的身份侧证据。下一步
是把每个分离源拿去对 live speaker store 打分，同时不改变 speaker label、pending pool、
exemplar 或 timeline 决策。

### 方法

在 Auditus 下新增 shadow-only 的 `SeparatioOratorProbe` peer。它拥有专用 CAM++ encoder
和 Povey fbank frontend 来提取 source embedding，然后通过 `SpeakerVectorStore::peek_best()`
只读打分。runtime 处于 dual-encoder 模式时，probe 会把 CAM++ 与现有 WavLM-ECAPA embedding
拼接；WavLM 提取由共享 mutex 保护。实时 SAAS WavLM 路径也使用同一把锁，因此 ASR worker
不会和实时 speaker pipeline 争用 encoder scratch buffer。

`fusion_shadow` payload 现在包含 `src1.speaker` 和 `src2.speaker` 对象，字段包括 ready、
extracted、accepted、reason、speaker id、similarity、second-best similarity、margin、
thresholds 与 timing。该路径仍然由 `DEUSRIDET_AUDITUS_FUSION_SHADOW=1` opt-in 控制，仍然
不改变 `SpeakerTimeline` 或最终 transcript label。

	`separatio_orator_probe.{h,cpp}`。

speaker similarity、margin 与 acceptance。下一步应跑更大的 clip batch，把 runtime event stream
对 GT 做比较，然后再考虑把 fusion decision 晋升为实际 attribution。

## Step 18q — Runtime fusion shadow batch telemetry

### 背景

单 clip 和三 clip smoke 已经证明 schema 能跑通，但它们还不能回答分离源 speaker evidence 是否
稳定到足以进入生产 selector。下一步需要一个可重复的 runtime batch：先用已知说话人的 reference
预热/注册 live store，再回放 top 风险 overlap clips，并把 `fusion_shadow` stream 对 GT 时间线复盘。

### 方法

新增 `tools/auditus_runtime_shadow_batch.py`，包含两个子命令：

- `prepare`：从干净 GT reference segments、静音间隔和
	`logs/segment_homogeneity_clips_r3/clip_manifest.jsonl` 中的 top 风险 clips 生成
	16 kHz mono batch WAV。
- `summarize`：读取 WebSocket raw events，通过 reference birth 把 runtime speaker id 映射到 GT
	speaker，并把每个分离源分类为 `mapped_gt_hit`、`mapped_gt_wrong`、`accepted_unmapped` 或
	`abstain`。

该 summary 只作为 telemetry，不是 primary speaker-attribution benchmark。它使用直接 runtime events
让人工复盘可执行，同时保持“最终 benchmark 分数不能只是脚本结论”的规则。

第一版 r1 batch 使用 8 段短 reference 加 12 个风险 clips。它暴露了两个问题：summarizer 曾把
`speaker_id=0` 误当成缺失值；而短 reference 也不能稳定让 online store birth 出四个 speaker。
解析器现在会保留零号 id，speaker-id mapping 也优先使用 reference 附近的 `speaker new` birth，
不再把 timeline context-fill label 当成真实注册来源。

shadow speaker margin 默认值从 `0.05` 收紧到 `0.055`（仍可通过
`DEUSRIDET_AUDITUS_FUSION_SPK_MIN_MARGIN` 覆盖）。对 r1 捕获 evidence 做离线 what-if 后，这是一处
最小改动：能去掉唯一的边界 false accept，同时不大面积吞掉有用命中。

### 结果

## Step 18r — Model residency budget guardrail

### 背景

runtime Auditus 工作给旧内存预算补上了一个重要现实条件：dual speaker 模式开启时，speaker stack
已经不再是 CAM++-only。`awaken` 默认配置 WavLM-ECAPA；ASR 在 speaker-only benchmark 运行中仍由
`DEUSRIDET_TEST_WS_ENABLE_ASR=1` gate 控制，MossFormer2 仍保持 lazy-loaded。

### 结果

更新 `.github/copilot-instructions.md` 和中英 Machina RFC，把 model residency 写成显式规则。新的生产
常驻估计把 LLM、ASR、TTS、CAM++ + WavLM-ECAPA，以及小型 VAD/enhancement sidecars 一起计入，
OS/CUDA 开销前约 47.0 GB，剩余约 13-14 GB 给 KV Cache、activations、递归态、长期记忆 hot set 和
shadow probes。

记录下来的规则很简单：新增大分配前必须说明它是 always-resident、env-gated、lazy 还是 offline-only；
只计算被选中的模型路径，不把整个模型目录当成运行时常驻；在 Tegra 上不能只依赖 `cudaMemGetInfo`
做预算决策。

## Step 18s — Runtime fusion shadow arbitrium gate

### 背景

runtime shadow 路径已经有 source ASR 和只读 speaker evidence，但仍缺一个保守的解释层。生产晋升
不能只看高 similarity：speaker store 必须稳定，矛盾必须可见，而且 shadow evidence 在通过回归前
不能拥有最终决策权。

### 方法

扩展 `fusion_shadow` 的 source speaker evidence，新增 `stable`、`exemplar_count`、`match_count`
以及稳定度阈值。默认阈值为 `DEUSRIDET_AUDITUS_FUSION_STABLE_MIN_EXEMPLARS=1` 和
`DEUSRIDET_AUDITUS_FUSION_STABLE_MIN_MATCHES=2`，两者都可以通过环境变量覆盖。

每条 `fusion_shadow` event 现在包含 shadow-only 的 `arbitrium` 对象。它不会修改 timeline label，
并且始终报告 `authoritative=false`；动作只是解释性候选，例如 `observe_only`、`support_timeline`、
`support_single_speaker`、`propose_split` 和 `record_contradiction`。当稳定 source 与现有 timeline
冲突时，现在会明确记录 contradiction，而不是用 support 类语言遮盖。

batch summarizer 也同步扩展，会报告 arbitrium action/reason 分布和 stable source evidence，后续调参
可以直接复盘 shadow gate 行为。

### 结果

验证：

- CMake Tools build：通过。
- diagnostics：本轮涉及的 C++ 和 tool 文件无错误。
- 行数预算：`audio_pipeline.cpp` 427，`separatio_orator_probe.cpp` 291，
	`separatio_orator_probe.h` 81。
- WebUI / WebSocket 检查：`/` 200，`/js/app.js` 200，`/css/layout.css` 200，`/ws` 101。
- `python3 -m py_compile tools/auditus_runtime_shadow_batch.py`：通过。

runtime smoke 产物：

- `logs/fusion_arbitrium_smoke_r1/live_capture_r3/raw_events.jsonl`
- `logs/fusion_arbitrium_smoke_r1/live_capture_r3/hyp.jsonl`
- `logs/fusion_arbitrium_smoke_r1/runtime_shadow_batch_report_r3.md`
- `logs/fusion_arbitrium_smoke_r1/runtime_shadow_batch_report_r3.json`

r3 smoke 回放了一段 33.7 秒 batch，产生 520 条 WebSocket raw rows、8 条 transcript 和 8 条
`fusion_shadow` event。所有 raw rows 都能按 JSON 解析。最终 gate 分布为 `observe_only=7`、

### 结论

本阶段现在具备了最低限度的生产形态 gate：分离源 ASR、只读 speaker evidence、store 稳定度元数据、
显式 contradiction 记录，以及可持久化的 runtime telemetry。它仍然不做权威归因。下一步晋升前应使用
这个 gate 跑更大的 r2 shadow regression，并要求 speaker enrollment 稳定之后才考虑修改 timeline label。

## Step 18x — Strong pending registration guard
背景/方法：post-commit r2 replay 在 `logs/fusion_postcommit_r2/` 暴露第四个 speaker birth miss；registration margin guard 因两个既有数据库 speaker 过近而阻止新 pending speaker，即使 `pending_sim=0.992`。`SpeakerVectorStore` 现在只在 best existing database match 仍低于正常 match threshold 时，允许 `pending_sim>=0.95` 绕过该 guard，从而保留 duplicate protection 和 shadow-only fusion authority。
