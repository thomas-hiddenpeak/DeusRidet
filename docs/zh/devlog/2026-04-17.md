# DEVLOG — 2026-04-17

## Decode 算子融合 + INT4 Marlin Attention：113→89 ms/tok（提速 21%）

### 背景

前几个 session 优化了 prefill（FP16 GEMM 162 GB/s = 可达天花板 177 GB/s 的 92%）。
Decode 基线：113.4 ms/tok，使用 INT8 GEMV 做 attention 投影 + Marlin INT4 做 MLP，
CUDA Graph 约 1063 节点。

### 优化 1：DeltaNet 融合 Decode 内核（113→109 ms，-4.5 ms）

**问题**：DeltaNet decode 在 QKV 投影后使用 6 个独立小内核（repeat_interleave ×2、
compute_g_beta、l2norm_scaled、l2norm、deltanet_recurrent），其中 recurrent 内核
每层从全局内存读写 128×128 的状态矩阵（48 heads × 64KB = 3 MB/层）。

**方案**：复用 prefill 的 `deltanet_fused_head_kernel`（M=1），将 a/b 投影直接写入
`dn_qkv` 缓冲区的 offset 10240/10288 处。该内核将所有 6 步操作融合为单个内核，
并使用寄存器缓存 S[128]，消除中间全局内存读写。

**结果**：108.9 ms/tok，951 CUDA Graph 节点（原 1063）。输出正确。

### 失败实验：MLP 使用 GPTQ GEMV（已回滚）

**假设**：GPTQ GEMV（169 GB/s）应快于 Marlin（155 GB/s）。
**结果**：120.8 ms/tok — 严重退化（+12 ms）。Marlin 的持久化内核 + `cp.async`
双缓冲在 MLP 形状（K=5120/17408）上完胜 GPTQ GEMV 的简单直接加载方式。
教训：SM87 上 Marlin > GPTQ GEMV，所有 INT4 矩阵尺寸均如此。

### 优化 2：Marlin INT4 替代 INT8 做全部 Attention Decode（109→89 ms，-20 ms）

**问题**：INT8 GEMV 虽然带宽利用率高（95%），但读取的权重数据是 INT4 的 2 倍。
对于 M=1 decode（完全内存受限），数据量减半的收益远大于内核效率的微小差异。

**方案**：使用 prefill 已有的 Marlin INT4 权重（`quantize_attn_to_marlin`）做 decode。
- DN（48 层）：3 个 INT8 GEMV + 1 个 INT8 dual GEMV → 3 个 Marlin GEMM（merged qkv_ab、z、out）
- FA（16 层）：3 个 INT8 GEMV → 3 个 Marlin GEMM（q、merged kv、o）

**结果**：**89.0 ms/tok**（稳定），903 CUDA Graph 节点。Token 输出完全一致。
