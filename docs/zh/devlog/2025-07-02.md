# DEVLOG — 2025-07-02

## FP16 GEMM 内核优化：达到 cuBLAS 同等性能

### 背景

继续优化 attention 权重投影的 FP16 GEMM 自定义内核。起始点：0.93x cuBLAS（DN qkv
上 650μs vs 606μs）。通过 ncu 定位根因：A 矩阵 L2 重读（267 MB vs cuBLAS 159 MB），
原因是 B 的 SMEM padding 占用过多空间导致 L1 不足以缓存 A tile。

### 优化路径

**1. B SMEM XOR swizzle（无 padding 的 bank-conflict-free 方案）**

原始方案 B_PAD=8 → B_STRIDE=72 → 每 TB 68KB SMEM → L1 仅剩 27KB < A 4 阶段 32KB →
A 被 L1 驱逐 → L2 重读。替换为 XOR swizzle：`col_int4 ^ (row % 8)`，bank 冲突率
降至 0.05%，同时 B_STRIDE=64 → SMEM 64KB → L1 35KB → A tile 完整缓存。L2 流量从
267MB 降至 194MB（降 27%），DRAM 放大率仅 3.5%。

**2. ldmatrix.x2.trans 实验（放弃）**

尝试硬件转置加载 B fragment。编写诊断工具 test_ldsm_trans.cu 证明寄存器映射不兼容
[N,K] 源布局：ldmatrix 输出 N 和 K 维度互换。需要 [K,N] 源格式，无法通过 cp.async
从 [N,K] 全局内存实现。回退为手动 half2 加载 + swizzle 寻址。

**3. 智能 grid 调度（波平衡）**

当 n_tiles 不整除 MAX_CONCURRENT (32)时, 非持久化调度导致末波浪费。例如 [6144,5120]
80 tiles → 2.5 波 → 末波 50% 空闲。解决方案：不整除时启用持久化（grid 限制
为 32），配合 cp.async.ca 的 L1 A 缓存实现跨 tile A 复用。[6144,5120] 从    411μs
(0.92x) 提升至 375μs (1.01x)。

### 最终结果 — 全部 Attention 投影（M=64, Prefill）

| 形状 | cuBLAS (μs) | Custom (μs) | 比率 | 带宽 (GB/s) |
|------|-------------|-------------|------|------------|
| DN qkv [5120,10240] | 605.8 | 613.9 | 0.99x | 162.1 |
| FA q [5120,12288] | 725.3 | 734.0 | 0.99x | 162.5 |
| DN z [5120,6144] | 376.2 | 372.0 | 1.01x | 161.1 |
| DN/FA out [6144,5120] | 377.5 | 375.4 | 1.01x | 159.7 |
| FA kv [5120,2048] | 129.0 | 127.8 | 1.01x | 159.6 |

全 64 层加权：整体 0.2% 慢于 cuBLAS。5 个形状中 3 个超过 cuBLAS。有效带宽
159-162 GB/s（理论 175 GB/s 的 91-93%）。

### 关键发现

1. **B_PAD 代价超预期**：8 halves/row × 64 rows × 4 stages = 4KB SMEM → L1 减少
   8KB → A 缓存级联效应
2. **ldmatrix.x2.trans 有特定源布局要求**：源必须为 [K,N]（K 行,N 列），我们的
   [N,K] 权重布局不兼容
3. **波不平衡影响显著**：n_tiles 不整除 max_concurrent 时，半满末波可损失 8-20%
4. **cp.async.ca L1 缓存有效**：同 SM 两个 TB 共享 L1 A tile，大幅减少 L2 流量
