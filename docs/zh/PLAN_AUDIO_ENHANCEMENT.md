# 音频管道增强计划 — 全面评估

> 日期: 2025-07-22
> 状态: 草案 — 待Thomas审核
> 背景: v24d 基线说话人分离准确率中位数 60-65%
> 测试素材: 4个相似中国男性说话人, 60:15 分钟音频

## 摘要

三阶段增强方案，逐一解决说话人误识别的根本原因：
**信号质量 → 重叠感知 → 声源分离**。每个阶段独立可测试、可构建，
设有明确的通过/终止门限。

```
当前管道:
  [麦克风PCM] → 增益 → RMS → Silero VAD → 语音积累 → 说话人嵌入 → 追踪
                                                                ↑
                                                        (噪声劣化嵌入质量)
                                                        (重叠混淆单一嵌入)

增强管道:
  [麦克风PCM] → 增益 → RMS → [P0: FRCRN降噪] → Silero VAD → 语音积累
                                                                   ↓
                                  [P1: pyannote OSD] ← ← ← ← fbank/emb特征
                                       ↓ 重叠?
                                  是 → [P2: MossFormer2-SS] → 逐说话人PCM → 嵌入 × 2
                                  否 → 单说话人嵌入 (同现有流程)
```

---

## 阶段 0 (P0): FRCRN 语音增强（降噪）

### 目标

在所有下游处理（VAD、说话人嵌入、ASR）之前清洁PCM信号。这同时改善
所有环节 — 更干净的信号意味着更好的VAD决策、更锐利的说话人嵌入、
更准确的ASR转录。

### 模型: FRCRN_SE_16K

| 属性 | 值 |
|------|-----|
| 来源 | [ClearerVoice-Studio](https://github.com/modelscope/ClearerVoice-Studio) |
| 架构 | DCCRN = ConvSTFT → 双UNet（复数掩码估计）→ ConviSTFT |
| 权重大小 | ~161 MB (FP32), ~80 MB (FP16) |
| 许可证 | Apache-2.0 |
| 输入 | 原始PCM float32, 16kHz单声道 |
| 输出 | 增强PCM float32, 等长 |
| STFT参数 | win_len=640 (40ms), win_inc=320 (20ms跳), fft_len=640, hanning |
| 掩码类型 | 复数理想比率掩码 (cIRM), tanh激活 |
| ONNX兼容性 | ConvSTFT/ConviSTFT为Conv1d实现，UNet为标准Conv2d — 全部ONNX友好 |

### 关键决策

- **运行时**: 先用ONNX Runtime CPU EP（与Silero/FSMN一致），延迟过高再升级TRT
- **插入点**: `audio_pipeline.cpp` 第349→351行，RMS计算之后、Silero VAD之前
- **处理策略**: 增强发生在所有下游处理之前，VAD/说话人/ASR全部接收干净PCM

### 通过门限

- **通过**: 说话人准确率中位数提升 >2% 或可听到降噪效果且无伪影
- **终止**: 延迟 >100ms/块（管道不可用）或伪影劣化ASR质量

## 阶段 1 (P1): 学习型重叠检测

### 目标

用学习模型替换启发式重叠检测（`jitter > 0.15f && low_sim_count_ >= 1`），
直接分类音频帧为单说话人 vs 多说话人。当前启发式误报率高，遗漏大量真实重叠。

### 模型: pyannote/segmentation-3.0（推荐）

| 属性 | 值 |
|------|-----|
| 架构 | PyanNet (SincNet + LSTM + Linear) |
| 许可证 | MIT |
| 输入 | 10秒单声道音频 @ 16kHz → (1, 1, 160000) |
| 输出 | (num_frames, 7) 幂集编码 |
| 类别 | 非语音, spk1, spk2, spk3, spk1+2, spk1+3, spk2+3 |
| 权重大小 | ~5 MB |
| 帧分辨率 | ~16ms/帧 (10秒 → ~625帧) |
| 关键优势 | **单模型同时实现VAD + OSD + 说话人分割** |

### 幂集解码逻辑

```
对每帧:
  argmax(7类)
  if argmax ∈ {4,5,6} → 重叠
  if argmax ∈ {1,2,3} → 单说话人
  if argmax == 0 → 非语音
```

### 通过门限

- **通过**: OD精确度 >70% 且误报率 <10%
- **终止**: ONNX导出失败 → 退回到现有特征上训练小分类器

## 阶段 2 (P2): MossFormer2 语音分离

### 目标

当P1检测到重叠时，将混合音频分离为独立说话人流，从每个分离流独立提取
说话人嵌入。这是关键能力缺口 — 当前管道完全无法处理重叠说话人。

### 模型: MossFormer2_SS_16K

| 属性 | 值 |
|------|-----|
| 架构 | Conv1d编码器 → MossFormer2块 × 24 → ConvTranspose1d解码器 |
| 权重大小 | ~670 MB (FP32), ~335 MB (FP16) |
| 许可证 | Apache-2.0 |
| 输出说话人数 | 2（固定） |
| 处理段长 | 2秒（原生），长音频分段处理 |

### 内存策略

1. **FP16推理**: 降至335 MB
2. **延迟加载**: 首次检测到重叠时加载，之后常驻
3. **SSD换入**: 必要时从SSD加载（~170ms @ 2 GB/s）

### 说话人-声源分配

MossFormer2输出source1和source2，分配到哪个说话人是任意的（排列问题）。
解决方案:
1. 从两个源提取嵌入
2. 将两个嵌入与当前 `ref_emb_`（已知说话人）比较
3. 与ref_emb_相似度更高的源是当前说话人
4. 另一个源是重叠说话人
5. 在说话人数据库中匹配重叠说话人

### 通过门限

- **通过**: 合成重叠测试中两个说话人正确识别率 >60% 且分离延迟 <500ms
- **终止**: ONNX导出失败或内存超预算 → 研究更轻量级分离器

```
P0 (FRCRN)          P1 (OD)               P2 (MossFormer2-SS)
│                    │                     │
├─ ONNX导出          ├─ ONNX导出            ├─ ONNX导出
├─ C++封装           ├─ C++封装             ├─ C++封装
├─ 管道集成          ├─ 管道集成            ├─ 管道集成
├─ 独立测试          ├─ 独立测试            ├─ 配合P1测试
│                    │                     │
▼                    ▼                     ▼
门限0                门限1                  门限2
```

### 分版发布

1. **v25a**: 仅P0 — FRCRN降噪，测量基线改善
2. **v25b**: P0 + P1 — 添加重叠检测，测量OD准确率
3. **v25c**: P0 + P1 + P2 — 完整管道含分离，测量端到端准确率提升

每个版本独立提交、基准测试、DEVLOG记录。

## 待Thomas确认的问题

1. **FRCRN块大小**: 逐管道块（~30ms）处理还是积累到1-3秒后处理？
   短 = 低延迟，长 = 更好增强质量。

2. **OD粒度**: pyannote处理10秒窗口。5秒跳步（50%重叠）可接受吗？

3. **分离触发**: OD检测到重叠时，分离范围？
   (a) 整个语音段, (b) 仅重叠区域, (c) 重叠区域 + 1秒上下文？

4. **模型准备**: 你自己运行ONNX导出，还是我写完整导出脚本包？

5. **优先级**: 先启动P0（仅降噪），还是P0+P1并行启动？
